# 代收

## Term（术语）
代收 (Collection / Pay-in)

## Domain（所属领域）
第三方支付系统 (Third-party Payment System)

## Definition（标准定义）
代收是指第三方支付机构或商业银行接受**商户 (Merchant)** 或**债权人**的委托，依据用户（持卡人/付款人）的事先授权协议，直接从用户的银行账户、电子钱包或其他资金账户中发起扣款指令，并将资金结算给商户的资金流转过程。在技术实现上，通常涉及签约（Signing）、鉴权（Authentication）、扣款（Deduct）和清算（Clearing）等环节。

## Core Characteristics（核心特征）
1.  **基于授权 (Authorization-based)**：
    交易发起的必要前提是用户与商户/支付机构签署了代收协议（如实名验证、绑定银行卡、签署代扣协议），授权商户在特定条件下直接发起扣款。
2.  **商户发起 (Merchant-initiated)**：
    与网关支付（用户主动操作）不同，代收交易通常由商户系统在后台发起请求，无需用户在支付时刻进行即时确认（部分高风险场景除外）。
3.  **批量与自动化 (Batch & Automated)**：
    支持高并发的批量指令处理，适合周期性或触发式的自动化扣款场景。
4.  **资金归集 (Fund Consolidation)**：
    资金流通常是从多个分散的付款人账户流向一个集中的商户结算账户。

## Typical Use Cases（典型使用场景）
1.  **公共事业缴费**：
    水、电、燃气费的自动定期扣缴。
2.  **订阅制服务**：
    视频会员自动续费、SaaS 软件按月扣款、云服务资源按量计费。
3.  **信贷还款**：
    信用卡账单自动还款、消费金融分期自动代扣。
4.  **保险费缴纳**：
    长期寿险或车险的定期保费划扣。

## Related Concepts（相关概念）
*   **Withholding (代扣)**：代收的另一种通俗称呼，常用于强调“扣款”动作。
*   **Agreement Payment (协议支付)**：指基于协议授权的支付方式，代收是协议支付的一种典型应用形式。
*   **Recurring Payment (周期性支付)**：代收常用于实现周期性支付，但代收也可以是单次的。

## Common Misunderstandings（常见误解）
*   **误解 1：代收等同于快捷支付**。
    *   **澄清**：快捷支付通常需要用户在支付时输入密码或验证码进行确认（即时验证），而代收更多强调基于事先协议的后台自动扣款，支付时往往无感。
*   **误解 2：商户可以随意扣款**。
    *   **澄清**：代收受到严格的风控限制，包括单笔限额、单日限额、扣款频率等。且必须有明确的用户授权协议，否则属于违规行为。

## Notes / Constraints（适用条件或限制）
1.  **渠道覆盖与限额**：
    不同银行或支付通道对代收业务的支持力度不同，B2B 和 B2C 的限额差异巨大，系统设计时需考虑**路由策略 (Routing Strategy)** 以适配不同通道的限额和成功率。
2.  **失败重试机制**：
    由于余额不足或网络原因导致扣款失败是常态，系统需设计合理的**重试策略 (Retry Policy)**（如指数退避），避免短时间高频重试触发风控。
3.  **幂等性 (Idempotency)**：
    代收接口必须严格保证幂等性，防止因网络超时等原因导致用户被重复扣款。
4.  **合规性**：
    需遵循监管机构关于“断直连”及代收业务规范的要求，明确适用场景（如近年来监管严禁将代收用于P2P理财、即时消费等高风险场景）。
